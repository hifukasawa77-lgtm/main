# GALACTIC ASSAULT 要件定義書

## 1. 概要

`GALACTIC ASSAULT` は、ブラウザ上で動作する横スクロール型スペースシューティングゲームである。プレイヤーは宇宙戦闘機を操作し、通常敵・隕石・ステージボスを撃破しながら全 7 ステージのクリアを目指す。

本要件定義では、既存実装をベースに、グラフィック品質を GPT-Image 2 生成アセットで大幅に向上させることを主目的とする。

## 2. 目的

- ポートフォリオサイト内で遊べるブラウザゲームとして、第一印象の品質を高める。
- 既存の軽量な HTML / CSS / JavaScript 構成を維持しつつ、ゲーム内スプライトの見栄えを改善する。
- PC とスマートフォンの両方で、追加インストールなしにプレイ可能な状態を保つ。
- サイバーパンク過多ではない、ダークでモダンなスペースオペラ調のビジュアルへ更新する。

## 3. 対象範囲

### 対象

- `galactic-assault.html`
- プレイヤー機、敵機、隕石、タレット、パワーアップカプセル、各ステージボスのスプライト
- Canvas 上の描画品質
- 既存ゲームプレイを壊さない範囲での演出改善

### 対象外

- ゲームエンジンやビルドツールの導入
- フレームワーク化
- ステージ構成やゲームルールの大幅変更
- オンラインランキング、ログイン、課金機能

## 4. 想定ユーザー

- ポートフォリオ訪問者
- PC ブラウザで手軽にゲームを試したいユーザー
- スマートフォンで短時間プレイしたいユーザー
- 制作者のフロントエンド / ゲーム制作スキルを確認したい閲覧者

## 5. 基本仕様

- ゲーム形式: 横スクロール型 2D シューティング
- 実行環境: モダンブラウザ
- 実装方式: 単一 HTML + CSS + JavaScript
- 描画方式: Canvas API
- 音声方式: Web Audio API
- 画面サイズ: 480 x 540 Canvas
- 操作方式: キーボード / タッチ操作
- ステージ数: 7
- 初期残機: 3

## 6. 機能要件

### 6.1 タイトル / 状態遷移

- タイトル画面を表示すること。
- ショット入力でゲームを開始できること。
- ステージ開始前にステージ番号、ステージ名、ボス名を表示すること。
- 全ステージクリア時はエンディング画面へ遷移すること。
- 残機が 0 になった場合はゲームオーバー画面へ遷移すること。
- ゲームオーバー / エンディング画面からタイトルへ戻れること。

### 6.2 プレイヤー操作

- キーボード操作に対応すること。
  - 移動: 矢印キー / WASD
  - ショット: Z / Space
  - パワーアップ実行: X
  - ポーズ / 再開: Enter
  - 隠しコマンド: 上上下下左右左右XBで、開始時から全パワーアップを有効化する。
- タッチ操作に対応すること。
  - 仮想 D-Pad で移動
  - SHOT ボタンで攻撃
  - POWER ボタンでパワーアップ実行
- プレイヤーは Canvas 範囲外へ移動できないこと。

### 6.3 攻撃

- 通常ショットを発射できること。
- レーザーレベルに応じて弾サイズ、威力、連射性能が変化すること。
- MISSILE パワーアップにより、斜め下方向へ投下されるミサイルを発射できること。
- DOUBLE パワーアップにより、前方と斜め上の 2 方向へ通常弾を発射できること。
- レーザー取得後、ショット長押しによりチャージ攻撃を発動できること。
- オプション取得時、追従ユニットから補助弾を発射できること。

### 6.4 パワーアップ

- 敵撃破数に応じてパワーアップカプセルが出現すること。
- 赤いパワーアップカプセル取得時、パワーアップカーソルが右へ 1 つ移動すること。
- POWER 入力で以下の効果を発動できること。
  - SPEED UP: 自機の移動速度を段階的に上昇させる。
  - MISSILE: 斜め下方向へミサイルを投下する。
  - DOUBLE: 前方と斜め上方向へ通常弾を撃つ。
  - LASER: 敵を貫通する前方ビームを撃つ。2 段階目はツインレーザーにする。
  - OPTION: 自機の動きを追従する分身を追加する。
  - SHIELD: 敵弾や体当たりを複数回防ぐ。
- 各パワーアップには上限値を設けること。

### 6.5 敵

- 通常敵を一定間隔で出現させること。
- ステージ 2 以降は隕石を出現させること。
- 既存実装と比較して、道中に出現するザコ敵の総数を約 2 倍に増やすこと。
- ザコ敵はステージテーマに合わせて外見、移動、攻撃パターンを変えること。
- ステージ固有ギミックから発生する攻撃も、敵弾または障害物として扱うこと。
- 各ステージ後半でボスを出現させること。
- 敵は撃破時にスコアを加算すること。
- 敵または敵弾との接触時、プレイヤーへダメージを与えること。

### 6.6 ステージ / ボス

- 全 7 ステージを実装すること。
- 各ステージの道中は現状より大幅に長くし、ボス出現までにステージ固有の地形・ギミック・ザコ敵を十分に体験できる長さにすること。
- 目安として、各ステージの道中は 60 秒から 90 秒程度、ボス戦を含めて 1 ステージ 90 秒から 150 秒程度のプレイ時間を想定すること。
- ボス出現前の道中には、単調な星空スクロールだけではなく、ステージ固有の背景オブジェクト、固定砲台、障害物、演出を配置すること。

| Stage | Name | Boss |
| --- | --- | --- |
| 1 | VOLCANO | TWIN DRAGON |
| 2 | MOAI | MOAI TRIPLE GOD |
| 3 | PYRAMID | TUTANKHAMUN |
| 4 | RUINS | ALIEN OVERLORD |
| 5 | ALIEN BODY | LARGE AMOEBA |
| 6 | COSMOS | SPACE FORTRESS |
| 7 | FORTRESS | MEGA ROBOT |

- Stage 1 `VOLCANO`
  - 火山地帯を舞台とすること。
  - 背景または地形として、所々に火山を配置すること。
  - 火口から火山弾が噴出し、プレイヤーへの障害物または敵弾として機能すること。
  - 火山弾は放物線、散弾、連続噴出など複数パターンを持たせること。
  - ザコ敵として火の鳥を出現させること。
  - 上下の地形に砲台を配置し、プレイヤーへ向けて弾を撃たせること。
  - ステージ後半に、上側地形から垂れ下がる逆さ火山を配置し、大量の噴石を下方向へ噴出させること。
- Stage 2 `MOAI`
  - イースター島を舞台とすること。
  - 所々にモアイ像を並べて配置すること。
  - モアイ像は口からリング状レーザーを発射すること。
  - リング状レーザーは視認しやすい円形または輪状の弾として描画すること。
  - ステージに適したザコ敵を追加で 3 種以上用意すること。
- Stage 3 `PYRAMID`
  - ピラミッド内部を舞台とすること。
  - 石壁、通路、古代装置、罠など、内部ステージらしい背景と地形を配置すること。
  - ピラミッドに適したザコ敵を出現させること。
  - 例: スカラベ型ドローン、石棺タレット、古代機械兵、ミイラドローン、アヌビス型ロボット、スカラベ爆撃機。
- Stage 4 `RUINS`
  - 古代遺跡を舞台とすること。
  - 崩れた柱、石碑、浮遊する遺物、封印装置などを配置すること。
  - 遺跡に適したザコ敵を出現させること。
  - 例: 浮遊石像、遺跡守護ドローン、石版シールド敵、結晶弾を撃つ砲台、石像バット、浮遊オベリスク、結晶蛇。
- Stage 5 `ALIEN BODY`
  - エイリアンの体内を舞台とすること。
  - 有機的な壁、脈動する背景、粘液、管状の地形を配置すること。
  - 体内に適したザコ敵を出現させること。
  - 例: アメーバ、触手、細胞型敵、寄生虫型敵、粘液弾を撃つ器官砲台、白血球型敵、ヒル型敵、胞子クラゲ。
- Stage 6 `COSMOS`
  - 宇宙空間を舞台とすること。
  - 星雲、小惑星帯、人工衛星、敵艦隊の残骸などを配置すること。
  - 宇宙に適したザコ敵を出現させること。
  - 例: 高速迎撃機、ドローンスウォーム、シールド艦、機雷、隕石群、円盤機、彗星獣、衛星砲台。
- Stage 7 `FORTRESS`
  - 敵要塞内部を舞台とすること。
  - 金属壁、隔壁、砲台、レーザーゲート、動力炉などを配置すること。
  - 要塞内部に適したザコ敵を出現させること。
  - 例: 防衛ドローン、固定レーザー砲台、重装甲兵器、誘導ミサイルポッド、警備ロボット、ブレードドローン、装甲ミサイルポッド、レーザーゲート砲台。
- 各ボスは固有の HP、移動、弾幕パターンを持つこと。
- ボス撃破時、ステージクリアとし次ステージへ進行すること。
- 最終ボス撃破時、エンディングへ遷移すること。

### 6.7 スコア / HUD

- 画面上部にスコア、ステージ名、残機を表示すること。
- 画面下部にパワーアップバーを表示すること。
- ボス戦ではボス HP ゲージとボス名を表示すること。
- チャージ中はチャージゲージを表示すること。

### 6.8 グラフィック

- プレイヤー、敵、ボス、アイテムは GPT-Image 2 生成アセットを使用して品質向上すること。
- ステージ固有の背景オブジェクト、ギミック、ザコ敵についても GPT-Image 2 生成アセットを使用すること。
- Stage 1 の火山、火口、火山弾、Stage 2 のモアイ像、リング状レーザー、Stage 3 以降のテーマ別ザコ敵は、個別に識別できる専用グラフィックを用意すること。
- スプライトは透過 PNG として Canvas に描画できること。
- 既存の黒背景 + シアン / パープル系アクセントと調和すること。
- 過度なネオングロウ、SF 都市風、サイバーパンク演出は避けること。
- 小さく描画しても識別しやすいシルエットを保つこと。
- 敵、地形、弾、背景の視認性を優先し、当たり判定が分かりにくくなる過度な装飾は避けること。
- 既存 SVG 版は必要に応じてバックアップまたは旧アセットとして残すこと。

### 6.9 サウンド

- Web Audio API により BGM を再生できること。
- ユーザー操作後に音声初期化を行い、ブラウザの自動再生制限に対応すること。
- ステージごとの雰囲気に合わせた BGM パターンを持つこと。
  - 火山: 高速で攻撃的なロック調。
  - イースター島: 低速で儀式的なリズム。
  - ピラミッド: 砂漠・古代遺跡を想起させる旋律。
  - 遺跡: 不穏で機械的なリズム。
  - エイリアン体内: 不気味で有機的な低音中心。
  - 宇宙: 壮大で浮遊感のあるテンポ。
  - 要塞: 最終面らしい高速で緊張感のある曲調。
- ゲームオーバーまたはエンディング時に BGM を停止すること。

## 7. 非機能要件

### 7.1 パフォーマンス

- 追加ビルドなしで GitHub Pages 上に配置できること。
- Canvas 描画は通常プレイ時に体感上滑らかであること。
- 画像アセットはブラウザ読み込みに支障がない容量に抑えること。
- 画像読み込み失敗時もゲームが停止しないこと。

### 7.2 互換性

- Chrome / Edge / Safari / Firefox の最新安定版で動作すること。
- PC とスマートフォンの両方で操作可能であること。
- 外部 CDN やネットワーク依存なしでゲーム本体が動作すること。

### 7.3 保守性

- フレームワークは使用しないこと。
- ゲームロジックは既存の素の JavaScript 構成を維持すること。
- アセット参照は定数化し、差し替えしやすい構造にすること。
- 生成アセットは用途が分かるファイル名で管理すること。

### 7.4 デザイン制約

- リポジトリ全体のポートフォリオデザイン方針に合わせること。
- 黒背景 + シアン / パープル系アクセントを基本とすること。
- サイバーパンクテーマは禁止すること。
- ゲーム画面内の文字は可能な範囲で日英バイリンガル対応を考慮すること。

## 8. 受け入れ条件

- ブラウザで `galactic-assault.html` を開き、タイトル画面が表示されること。
- キーボードで移動、ショット、パワーアップが操作できること。
- スマートフォン相当のタッチ操作で移動、ショット、パワーアップが操作できること。
- GPT-Image 2 生成スプライトが Canvas 上で透過表示されること。
- 主要スプライトにマゼンタ等の抜き残りが目立たないこと。
- 7 ステージ構成とボス出現が維持されていること。
- ゲーム開始、ステージ進行、ゲームオーバー、エンディングの状態遷移が動作すること。
- JavaScript 構文エラーが発生しないこと。
- `.edge-test-profile/` がコミット対象に含まれていないこと。
