# /self-improve — 再帰的自己改善ループ

セカンドブレインは「学びを**蓄積**」する。このスキルは蓄積した学びを**ハーネス自身の指示**（エージェント定義・CLAUDE.md・スキル・フック）へ**反映**することで、ループを閉じる。

```
   [蓄積] obsidian-vault/ に学び・決定を記録（/second-brain）
       │
       ▼
   [想起] SessionStart hook が MOC・知見インデックス・直近Dailyを毎回投入
       │
       ▼
   [反映] /self-improve が再利用可能な教訓を恒久ルールへ昇格 ← このスキル
       │
       ▼
   次セッション/パイプラインが恒久的に賢くなる（再帰）
```

「想起」だけでは学びは毎回読み直されるコストのまま残る。価値ある教訓は**指示そのもの**に昇格させて初めて自動で効く。それがこのスキルの役割。

## いつ使うか
1. セッションの区切り（大きめタスク完了時）— その日の学びに昇格すべきものがないか点検する
2. `04-Knowledge/` の知見や `01-Daily/` の「学んだこと」が一定量たまったとき
3. **同種のミス・レビュー指摘・不合格が再発したとき**（＝ルール化されていない証拠。最優先で昇格）
4. ユーザーが「自己改善して」「学びを反映して」と指示したとき

## 手順

### 0. ヘルスチェック（Lint）— 機械的に検出できる劣化を先に潰す
振り返りの前に `bash .claude/skills/self-improve/harness-lint.sh` を実行する。以下を一括検査する（過去に手作業で繰り返した監査の機械化）:
- スキルの `SKILL.md` 存在（`Skills.md` 等の誤名・欠落でサイレント死蔵していないか）
- agents/skills の frontmatter 妥当性（1行目`---`・`name:`・`description:`・閉じ`---`）
- 課金ガードの2段ゲート同期（settings.json matcher == accounting-guard.sh の BILLING_RISK_TOOLS）
- 色ドリフト（CLAUDE.md禁止のマゼンタ/原色ネオンが**指示として**残っていないか）
- 具体値ドリフト（予算上限/採点閾値/作業ブランチが正の単一ソースと再掲先で一致するか）
- 主要参照パスの実在

`✗` が出たら手順2〜4で是正する。`✅`（exit 0）なら振り返りへ進む。新しい機械検査が必要になったらこのスクリプトに検査を追記して恒久化する。追記時はbash罠に注意（`grep | while` のサブシェルではフラグが親に伝播しない→here-stringで回す／検査スクリプト自身が検出パターン文字列を含むため自己除外する。詳細: [[harness-maintenance-patterns]]）。

### 1. 振り返り（Reflect）— 昇格候補を集める
- `obsidian-vault/04-Knowledge/*.md` と直近の `01-Daily/*.md`（最新3〜5件）を読む。`03-Decisions/` の最新も確認する。
- 各学びを次の観点で分類する:
  - **再発・反復しているか**（同じ注意書きが複数ノートに出る → 強い昇格候補）
  - **再利用可能か**（特定の1回限りでなく、今後も効く一般則か）
  - **既にルール化済みでないか**（CLAUDE.md・該当エージェント定義を grep して重複を避ける）
- 一回限りの事象・特定タスク固有のメモは昇格しない（Vaultに置いたままでよい）。

### 2. 宛先の特定（Locate）— どの指示に書くか
学びの性質ごとに最も狭く効く場所へ置く（広すぎる場所に書かない）:

| 学びの種類 | 還元先 |
|---|---|
| 特定エージェントの振る舞い・手順・チェック観点 | `.claude/agents/<該当>.md` |
| プロジェクト横断の方針・禁止事項・コーディング規約 | `CLAUDE.md` の該当セクション |
| 再現性のある作業手順・パイプライン | 既存スキル `.claude/skills/<該当>/SKILL.md` または新規スキル |
| セッション開始時に毎回効かせたい前提 | recall hook / CLAUDE.md |
| 自動で機械的に強制したい制約 | `.claude/hooks/` ＋ `settings.json`（hookはLLMを呼べない単純判定のみ）|

例: 「新規innerHTML描画ではそのスコープからescHtmlが見えるか必ず確認」→ `security.md` と `code-generator.md` のチェック観点に追記。

### 3. 反映（Propagate）— 差分を入れる
- **追加型の低リスクな指示**（チェック観点・注意書きの追記）は Edit で直接反映してよい。
- **既存ルールを変える/振る舞いを変える**変更は、適用前に必ず `AskUserQuestion` で深澤に確認する（後で取り消しにくいため）。承認されたら ADR 化する（手順4）。
- 反映は**差分追記**で行う。ファイル全体の書き換え禁止（CLAUDE.md のコンテキスト節約ルールに従う）。
- 重複を作らない。既存の記述を**強化/具体化**できるなら新規追記より優先する。

### 4. 記録（Record）— 何を昇格したかをVaultへ書き戻す
- 昇格した内容を `01-Daily/YYYY-MM-DD.md` の「意思決定」または「学んだこと」に1〜2行で記録する。
- ルール/振る舞いの変更を伴う場合は `03-Decisions/NNNN-スラッグ.md`（連番は既存最大+1、テンプレ `Templates/decision-note.md`）を作成し、`related` に元の知見ノートを wikilink で結ぶ。
- 元の知見ノート末尾に「→ <還元先> に昇格済み（YYYY-MM-DD）」と追記し、二重昇格を防ぐ。
- 必要なら `MOC.md` にリンクを追加（孤立させない）。

### 5. 検証（Verify）
- 編集したエージェント定義/スキルの YAML frontmatter が壊れていないか確認する。
- recall hook を編集した場合は `bash .claude/hooks/second-brain-recall.sh` を実行して出力が壊れていないか確認する。
- コミットは CLAUDE.md の Git ルール（作業ブランチ・日本語メッセージ可・`.edge-test-profile/` 混入チェック）に従う。

## 原則・ガードレール
- **狭く効かせる**: 学びは最も限定的に効く場所へ。CLAUDE.md を肥大化させない。
- **重複させない**: 昇格前に必ず grep で既存ルールを確認する。
- **取り消しにくい変更は確認を取る**: 振る舞い変更は `AskUserQuestion` → ADR。
- **機微情報を書かない**: APIキー・パスワード等は還元先にも書かない（公開リポジトリ）。
- **役割分担を守る**: Vault=個人の知的資産、PMO=ステークホルダー向け進捗。このスキルはVault→ハーネス指示への一方向の昇格のみ扱う。
- **ループの健全性**: 反映後は必ず手順4で記録し、想起→反映が二重に走らないようにする。
