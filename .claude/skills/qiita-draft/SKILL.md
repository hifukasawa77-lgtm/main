# /qiita-draft — 技術記事の下書き

## 手順

1. **題材の特定**: 何を書くか（このリポジトリの実装・ハマりどころが最有力。`04-Knowledge/` の知見ノートは記事のタネ）
2. **過去記事の確認**: `docs/` 内の既存Qiita記事の文体・構成を踏襲する
3. **下書き作成**: 出力先は `docs/qiita-<スラッグ>.md`
4. **公開前チェック**（下記）を通してから深澤へレビュー依頼

## 記事構成テンプレ

```markdown
# タイトル（結果や数字を入れる: 「CanvasだけでトップビューRPGを作った」等）

## はじめに（つかみ・3行）
何を作った/解決したか。読者が得るもの。デモ/リポジトリへのリンク。

## 背景・課題
なぜやったか。既存手法の何が問題だったか。

## 実装
コア部分のみコードで見せる（全文は貼らずリポジトリへ誘導）。
手順は番号付き。スクショ/GIFがあると強い（test-screenshots/ を活用）。

## ハマりどころ
最低1つ書く（記事の価値の中心。04-Knowledge/ から引く）。

## まとめ
できたこと・できないこと・次にやること。
```

## コード引用ルール
- 引用は**スニペットのみ**（20〜40行目安）。ファイル全文を貼らない
- 本リポジトリのコードは自作なので引用自由。ただし外部由来のコード/素材が混ざる場合は legal-checker の確認を通し、ライセンス表記を付ける
- 記事内のURLは本番URL（`https://hifukasawa77-lgtm.github.io/main/...`）を使う

## 公開前チェック
- [ ] APIキー・Webhook URL・メールアドレス等の機微情報が本文/コード/スクショに写り込んでいない
- [ ] スクショに個人情報（ブックマークバー・通知等）が写り込んでいない（ヘッドレス取得の `test-screenshots/` なら安全）
- [ ] 事実の検証: 記事中の「動く」コードは /dynamic-test を通した状態
- [ ] タイトル・見出しだけ読んで内容が伝わるか
