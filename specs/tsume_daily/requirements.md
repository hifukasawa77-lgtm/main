## 1. 背景・目的

### 背景（市場調査レポートより）
- 対局将棋・問題集型詰将棋アプリは飽和（将棋ウォーズ800万人、みんなの詰将棋1200問等）。正面参入は勝ち目なし
- 一方「1日1問」型の習慣化詰将棋（Wordle型デイリー＋ストリーク＋シェア）は既存アプリの習慣設計が弱く空白ニッチ
- デイリーパズルはUA予算不要で長期リテンションが取れ、個人開発と相性が良い
- 既存資産 `shogi.html` の盤面・駒ロジック（移動計算・王手判定・打ち駒制約）が流用可能で開発規模は小

### 目的
1. Wordle型の「1日1問＋ストリーク＋結果シェア」詰将棋をWeb（GitHub Pages）で公開し、需要を検証する
2. シェア機能による自然流入（バイラル）でゼロ予算の集客導線を作る
3. 本フェーズの利用データ（継続率・シェア率）を、フェーズ2（Capacitor＋AdMobによるAndroid展開）の投資判断材料とする

## 2. ターゲットユーザー
- **メイン**: 将棋アプリ利用経験のある国内カジュアル層（将棋ウォーズ級〜初段未満）。スキマ時間に1問だけ解きたい層
- **サブ**: Wordle系デイリーパズル愛好者（日課＋シェア文化に親和性）、海外の将棋学習者（英語UI対応）
- **利用環境**: モバイルブラウザ中心（タッチ操作）、PCブラウザも対応

## 3. 機能要件（MoSCoW法）

### Must（MVP必須）
| ID | 機能 | 概要 |
|---|---|---|
| M1 | 日替わり問題 | 日付シードによる決定論的自動生成。全ユーザーが同一日（JST基準）に同一問題。1手詰または3手詰 |
| M2 | 問題自動生成 | 完全クライアントサイドで生成（著作権リスク回避のため既存問題集を一切使用しない）。詰み解の存在・唯一性を内蔵ソルバーで検証 |
| M3 | 解答プレイ | タッチ/クリックで攻方の駒を操作。受方（玉方）応手は自動（最長抵抗）。ミス上限5回、超過でその日は失敗 |
| M4 | ストリーク・統計 | localStorage保存: 現在連続日数 / 最長連続日数 / 挑戦日数 / クリア日数 / クリア率 / 平均ミス数 |
| M5 | 結果シェア | Wordle風絵文字グリッド＋テキストをクリップボードコピー。X共有（intent URL） |
| M6 | 日英バイリンガルUI | JA/EN切替トグル。設定をlocalStorageに永続化 |
| M7 | モバイル対応 | タッチ操作・レスポンシブ（最小幅360px）。盤面はビューポートに収まること |
| M8 | 1日1問制御 | 当日分クリア/失敗後は結果画面＋次回問題までのカウントダウン表示。同日の再挑戦不可（リプレイ閲覧は可） |

### Should（MVPに入れたいが、工数次第で次版）
| ID | 機能 | 概要 |
|---|---|---|
| S1 | ヒント機能 | 1日1回、動かすべき駒をハイライト。使用有無を統計に記録（フェーズ2のリワード広告の布石） |
| S2 | 解答リプレイ | クリア/失敗後に正解手順をアニメーション再生 |
| S3 | 統計モーダル | Wordle風の統計サマリー画面（ストリーク・分布表示） |

### Could（余裕があれば）
| ID | 機能 | 概要 |
|---|---|---|
| C1 | ミス分布グラフ | ミス回数別クリア分布の棒グラフ |
| C2 | 過去問題アーカイブ閲覧 | 過去日付の問題を練習モードで閲覧（統計には不算入） |

### Won't（フェーズ1ではやらない）
- 広告・収益化（AdMob等）→ フェーズ2（Android展開）のスコープ
- サーバーサイド機能・ランキング・アカウント（完全クライアントサイドを維持）
- 5手詰以上の高難度問題
- AI解説（提案3の段階導入オプションとして将来検討）
- プッシュ通知

## 4. 非機能要件

| 分類 | 要件 |
|---|---|
| パフォーマンス | 問題生成は通常500ms以内、最悪3秒以内（中級モバイル端末想定）。超過時は同梱フォールバック問題バンクに切替 |
| 互換性 | iOS Safari / Android Chrome / PC Chrome・Edge・Firefox の各最新2バージョン |
| オフライン | 初回ロード後はネットワーク不要（外部API・外部サーバーなし） |
| セキュリティ | XSS対策（innerHTML への外部由来文字列挿入禁止）。有料APIキー不使用。外部送信データなし |
| データ保全 | localStorageスキーマにバージョン番号を持たせ、将来のマイグレーションに備える。破損時は安全に初期化 |
| アクセシビリティ | タップ領域44px以上。色のみに依存しない状態表示（選択枠線など） |
| 法務 | 問題は自動生成のみ（既存詰将棋作品の転載・改変を禁止）。フォールバック問題バンクも自前アルゴリズムの事前生成物に限る。**リリース前にLegal-Checkerチェック必須** |

## 5. デザイン要件（CLAUDE.md準拠）
- 黒背景＋アクセントカラー（シアン / パープル系）
- Glassmorphismカード、Canvasアニメーションパーティクル背景（既存 index.html / shogi.html のトーンを継承）
- **サイバーパンク的演出（ネオングロウ過多・SF都市風）は禁止**
- UIは日英バイリンガル表記

## 6. 制約条件
- フレームワーク・ビルドツール不使用。素のHTML/CSS/JS。シングルHTMLファイル `tsume_daily.html` で完結
- GitHub Pagesホスティング（静的配信のみ）
- 既存 `shogi.html` の駒移動・王手判定ロジックを流用（コピーして組み込み。shogi.html自体は変更しない）
- 作業ブランチ: `kai_001`

## 7. 成功指標（KPI）— 需要検証用
| 指標 | 目標値（公開後4週間） | 計測方法 |
|---|---|---|
| ユニーク訪問者 | 300 UU | GitHub Pages流入（簡易計測はフェーズ1では任意。X反応数で代替可） |
| D7継続率 | 15%以上 | フェーズ1ではサーバーレスのため直接計測不可。Xシェア投稿の継続観測で代替（注: 正確な計測はフェーズ2で実装） |
| シェア実行率 | クリア者の10%以上 | 同上（X上の `#詰将棋デイリー` 投稿数で観測） |
| 当日クリア率 | 70〜90%（難しすぎず簡単すぎず） | 自己テスト＋ユーザーフィードバック |

※ サーバーレス構成のため定量計測には限界がある。フェーズ2移行判断は「X上のシェア数・反応」を主指標とする。
