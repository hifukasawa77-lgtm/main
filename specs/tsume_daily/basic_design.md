## 1. システム構成

```
[ユーザーのブラウザ]
   └─ tsume_daily.html（シングルファイル / GitHub Pages静的配信）
        ├─ <style>  … Glassmorphism / 黒背景＋シアン・パープル
        ├─ <canvas> … 背景パーティクル（装飾。index.html系の演出を踏襲）
        ├─ <div id="board"> … 9x9 DOMグリッド盤面（shogi.html方式を流用）
        └─ <script>
             ├─ I18n            … 日英辞書・言語切替
             ├─ SeededRandom    … 日付シード決定論的乱数
             ├─ ShogiCore       … shogi.htmlから流用する駒・判定ロジック
             ├─ TsumeSolver     … 詰み探索（生成検証＋受方応手に共用）
             ├─ TsumeGenerator  … 日替わり問題自動生成
             ├─ FallbackBank    … 事前生成問題バンク（埋め込みJSON）
             ├─ GameController  … プレイ進行・ミス管理・1日1問制御
             ├─ Stats           … localStorage統計・ストリーク
             ├─ Share           … 絵文字グリッド生成・コピー・X intent
             └─ UI              … DOM描画・モーダル・トースト
```

- 外部通信なし（完全クライアントサイド）。外部CDNライブラリも使用しない（依存ゼロ）
- shogi.html は変更しない。必要関数をコピーして組み込む（単一ファイル方針のため import はしない）

### 盤面描画方式の設計判断（PM承認論点）
- CLAUDE.mdは「ゲーム系はCanvas APIのみで完結」とするが、流用元 shogi.html の盤面は **DOMグリッド方式**（div + CSS）
- 本設計では **盤面=DOMグリッド流用、背景演出=Canvas** のハイブリッドを採用する
  - 理由: (1) 実証済み資産の最大流用で開発規模最小化（提案1の根幹）、(2) タッチ判定・アクセシビリティがDOMの方が堅牢、(3) アクション性がなく60fps描画が不要
- ※ Canvas完結を厳守する場合は盤面のCanvas再実装が必要（工数増）。**深澤の判断を仰ぐ**

## 2. 画面構成・遷移

シングルページ内のビュー切替（display切替。URL遷移なし）。

```
[起動]
  └→ (A) タイトル/今日の問題ヘッダ
        │  ロゴ / #問題番号 / 「N手詰」/ 日付 / 言語トグル / 統計アイコン
        ▼
      (B) プレイビュー（メイン）
        │  盤面(9x9) / 攻方持ち駒 / ミスインジケータ(●○○○○) / ヒントボタン
        │  ├─ 当日未プレイ → そのまま開始
        │  └─ 当日プレイ済 → (D)へ直行
        ▼ クリア or ミス5回
      (C) 結果モーダル
        │  成否 / 正解手順 / 今日の絵文字グリッド / ストリーク🔥
        │  [シェア(コピー)] [Xでシェア] [リプレイ▶(Should)] [統計を見る]
        ▼
      (D) 完了ビュー
           結果サマリー / 次の問題までカウントダウン(hh:mm:ss) / シェアボタン再掲

      (E) 統計モーダル（どこからでも開ける / Should）
           挑戦数・クリア率・現在/最長ストリーク・ミス分布(Could)
```

### UI/UX設計の要点
- **盤面**: 画面幅に応じて `min(92vw, 480px)` の正方形。マス目はタップ領域44px以上を確保
- **操作フロー**: 駒タップ→移動可能マスをハイライト（シアン）→移動先タップで着手。持ち駒も同様。成り選択はモーダル（shogi.htmlの確認UI流用）
- **ミス時**: 盤面シェイク＋ミスインジケータ点灯（マゼンタ系は使わずパープルで統一）。盤面は着手前に巻き戻す
- **受方応手**: 攻方着手後 400ms ディレイでアニメーション着手（思考感の演出）
- **カラー**: 背景 #05060a 系 / アクセント シアン(#22d3ee系)・パープル(#a78bfa系) / Glassmorphismカード（半透明＋blur）。ネオングロウ過多は禁止
- **バイリンガル**: 全ラベルを `data-i18n` キー管理。駒表記は漢字（玉/飛/角…）＋ENモード時はツールチップ/凡例で補助

## 3. データ設計（localStorage）

キーはすべて `tsumeDaily.` プレフィックス。

### 3.1 `tsumeDaily.stats.v1`（統計・ストリーク）
```json
{
  "version": 1,
  "currentStreak": 5,
  "maxStreak": 12,
  "totalPlayed": 30,
  "totalCleared": 28,
  "totalMistakes": 41,
  "lastResultDate": "2026-06-11",
  "history": {
    "2026-06-11": { "cleared": true, "mistakes": 1, "timeSec": 45, "hintUsed": false, "plies": 3 }
  }
}
```
- `history` は直近400日分のみ保持（超過分は古い順に削除。集計値は累積フィールドに保存済みのため失われない）
- ストリーク判定: `cleared:true` の日がJST基準で連続している日数

### 3.2 `tsumeDaily.session.v1`（当日のプレイ状態）
```json
{
  "version": 1,
  "date": "2026-06-11",
  "state": "playing | cleared | failed",
  "mistakes": 2,
  "hintUsed": false,
  "startedAt": 1760000000000,
  "movesDone": 1
}
```
- 日付が変わっていたら破棄して新規作成（リロード・離脱対策の途中状態復元用）

### 3.3 `tsumeDaily.pref.v1`（設定）
```json
{ "version": 1, "lang": "ja" }
```

### 例外設計
- localStorage不可（プライベートモード等）→ メモリ上のオブジェクトで代替動作＋「記録が保存されません」バナー表示
- JSONパース失敗・version不一致 → 該当キーを初期化（確認なしで安全側に倒す）

## 4. 問題自動生成方式の設計（本プロダクトの中核）

### 4.1 方針: 「詰み形の逆算生成 ＋ ソルバー全数検証」
既存詰将棋作品を一切使わず、以下の手順でクライアントサイド生成する。

```
(1) シード決定     dateStr("YYYY-MM-DD" JST) → ハッシュ → PRNG(mulberry32)初期化
(2) 難度決定       月〜金: 1手詰 / 土・日: 3手詰（固定カレンダー制）
(3) 詰み形の構築   受方玉を盤上端域(1〜3段目)にランダム配置
                   攻方駒2〜4枚（金銀桂香飛角＋成駒）を玉周辺に配置
                   受方の守備駒0〜2枚を配置
(4) 1手(3手)巻き戻し 攻方の最終手を逆算で1手(2手)戻し、初期局面と攻方持ち駒を決定
                   ※巻き戻しの代わりに(3)の段階で「あとN手で詰む」局面を直接乱択し
                     ソルバーで適合判定するリトライ方式を基本実装とする（詳細設計参照）
(5) ソルバー検証   TsumeSolverで以下を全数探索で確認:
                   - ちょうどN手で詰む（N-2手では詰まない）
                   - 攻方の全着手が王手（詰将棋ルール）
                   - 初手が唯一解（複数初手解＝余詰は不採用）
                   - 打ち歩詰めを解に含まない
                   - 受方は盤上・攻方持ち駒以外の全駒を合駒に使える（標準ルール）
(6) 不適合なら     サブシードをインクリメントして(3)からリトライ（上限あり）
(7) リトライ上限超過 → FallbackBank（同アルゴリズムで事前生成し埋め込んだ問題群）から
                   日付シードで決定論的に選出
```

### 設計判断の理由
- **逆算（詰み形から作る）を採るのは**、乱数局面から前向き探索で「たまたま詰む」局面を探すより生成成功率が桁違いに高く、モバイルの性能予算（通常500ms）に収まるため
- **決定論性**: 同じ日付なら全端末で同一問題。これがWordle型シェア文化（「今日の#123、ノーミスだった」）の前提
- **唯一解の要求**: 詰将棋の品位とシェア時の公平性のため。生成効率が悪い場合の緩和（「初手のみ唯一」→「複数解許容」）は深澤判断とする
- **著作権**: 生成器・フォールバックバンクとも自前アルゴリズム産。Legal-Checkerには「生成方式の説明＋サンプル出力」を提出する

### 4.2 詰将棋ルールの実装範囲
| ルール | 対応 |
|---|---|
| 攻方は毎手王手 | ソルバー・プレイ判定とも強制 |
| 受方は全応手（玉移動・取り・合駒）を試す | ソルバーで全数探索。プレイ時は最長抵抗の応手を自動選択 |
| 受方持ち駒=残り全部 | 標準ルールどおり実装 |
| 打ち歩詰め禁止 | 攻方着手生成でフィルタ（**shogi.htmlのvalidMoves_dropには未実装のため新規追加**） |
| 行き所のない駒・二歩 | shogi.htmlのvalidMoves_dropを流用 |
| 千日手・無駄合 | 3手詰以内では実質発生しないため簡易対応（合駒は全列挙し探索深さで打ち切り） |

## 6. 主要コンポーネントの役割

| コンポーネント | 役割 | 入出力 |
|---|---|---|
| I18n | 言語辞書と切替 | `t(key)` → 表示文字列 |
| SeededRandom | 日付→決定論的乱数列 | `createRng(seed)` → `rng()` (0..1) |
| ShogiCore | 駒移動・王手・着手合法性（純粋関数群） | 盤面/手 → 合法手リスト・判定 |
| TsumeSolver | N手詰の全数探索・解手順の算出 | 局面 → 解ツリー or null |
| TsumeGenerator | 日替わり問題の生成・検証リトライ | dateStr → 問題オブジェクト |
| FallbackBank | 生成失敗時の埋め込み問題（自前事前生成） | index → 問題オブジェクト |
| GameController | 進行管理（着手→応手→判定→結果）、1日1問制御 | UIイベント → 状態遷移 |
| Stats | 統計の読み書き・ストリーク計算 | 結果 → localStorage |
| Share | シェア文面生成・クリップボード・X intent | 結果 → テキスト/URL |
| UI | DOM描画・モーダル・カウントダウン・パーティクル背景 | 状態 → 画面 |
