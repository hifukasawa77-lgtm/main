## エージェントハーネス設計

成果物作成は、必要に応じて以下の役割分担で進める。Codex ではユーザーが明示的に依頼した場合のみサブエージェントを使い、それ以外は同じ責務を通常の作業フローとして扱う。

### Planner

- 深澤から要件をヒアリングする
- 要件定義書 → 基本設計書 → 詳細設計書の順で仕様書を作成
- 必要に応じて市場調査・改善提案を行う
- 深澤の承認後、Designer / Generator 相当の作業へ仕様書を引き渡す

### Designer

- 主にグラフィックを担当する
- Planner の要件をもとに、外部ツール連携またはフリー素材の取得 / 加工で画像を制作する
- 生成した画像をリポジトリへ追加し、Generator 相当の実装で使える形にする

### Generator

- Planner の仕様書をもとにプログラム・ドキュメント等を実装する
- 実装完了後は Evaluator 相当の検証へ進める
- Evaluator から不合格を受けた場合は修正して再提出する
- 2 回以上同じ理由で不合格になった場合は深澤へ報告・判断を仰ぐ
- 実装規模が大きくタイムアウトが見込まれる場合は、作業をファイル単位またはページセクション単位に分割する

### Evaluator

- Generator の成果物を仕様書と照らし合わせて 100 点満点で採点する
- 合格基準: 80 点以上 かつ 仕様適合性 16 点以上（XSS 等は即不合格）
- 不合格時: 具体的なフィードバックを Generator へ返す
- 合格時: 深澤へ結果報告する
- コミットや push は、ユーザーが明示的に依頼した場合のみ行う

### フロー概要

```text
深澤 → [Planner] 要件定義・設計書作成
     → [Designer] グラフィック制作
     → [Generator] 実装
     → [Evaluator] 検証・採点
          ↓ 不合格
       [Generator] 修正・再提出 → [Evaluator] 再検証
          ↓ 合格
       深澤へ報告
```

## コンテキスト節約のルール

### ファイル読み込みの基本原則

- Read 前にできるだけ `rg` / `find` で対象行番号を特定する
- 大きなファイルは全体読み込みを避け、必要な範囲だけ読む
- 上限目安: 対象行の前後 200 行、`index.html` 等の巨大ファイルは前後 50 行
- `.claudeignore` 記載ファイルは不用意に全文読み込みしない

### エージェント間のデータ受け渡し

- Generator 相当の作業へは変更箇所のみを渡す（ファイル全体を渡さない）
- 形式: 「ファイル X の Y 行目付近を Edit tool で以下に変更」
- Evaluator 相当の確認は `git diff HEAD` を優先する
- エージェント間のコードブロックにファイル全体を貼ることを避ける

```bash
git diff HEAD        # 未コミット変更確認
git diff HEAD~1 HEAD # 直前コミットの確認
```

### Generator の出力形式

- コードは変更箇所スニペット（前後 10 行含む）を基本にする
- ファイル全体出力は禁止（「省略なし」ルールより本ルールを優先）

### プランファイルの管理

- 完了タスクは詳細を削除し、1 行サマリーに置き換える
- プランファイルは「現在未完了のタスク」のみ保持する
